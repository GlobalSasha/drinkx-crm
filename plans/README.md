Still open ‚Äî these need a database (correctness/data risk; can't be verified blind):
- **PERF-1 part B (phone-match query refactor)** ‚Äî the inbox phone fallback scans
  all workspace leads and normalizes in Python via `normalize_phone` (bare digits,
  RU 7/8‚Üí10). The indexed `phone_e164` column uses a DIFFERENT normalization

  before changing ‚Äî do NOT ship blind.
- **CORR-2 (contact dedup UNIQUE)** ‚Äî adding a UNIQUE constraint will FAIL the
  migration if duplicate contacts already exist in prod. Requires a dedup/backfill
