# API-контракт: отправка заявки с сайта в CRM

Документ для разработчиков сайтов/лендингов DrinkX.
Версия: Sprint 2.2 G2 (актуальна по состоянию на 2026-06-02).

---

## 1. Назначение

Каждая форма на сайте DrinkX должна создавать лид в CRM автоматически — без ручного
переноса. Для этого бэкенд сайта (или embed-скрипт CRM) вызывает один эндпоинт:
`POST /api/public/forms/{slug}/submit`. CRM создаёт входящий лид, фиксирует
источник (домен, UTM), направляет его менеджеру и уведомляет команду. Сайт получает
ответ немедленно — обработка лида в CRM происходит асинхронно и не блокирует ответ.

---

## 2. Эндпоинт

```
POST https://crm.drinkx.tech/api/public/forms/{slug}/submit
```

`{slug}` — уникальный строковый идентификатор формы (например, `main-landing`,
`b2b-horeca`). Slug выдаётся в CRM на странице **Настройки → Формы** при создании
формы. Каждый сайт или лендинг получает свой slug — это позволяет CRM отслеживать
источник заявки с точностью до формы.

---

## 3. Заголовки

| Заголовок | Обязательность | Значение |
|---|---|---|
| `Content-Type` | Всегда | `application/json` |
| `X-Form-Key` | Только для защищённых форм (S2S) | Секретный ключ формы из CRM |

**Об `X-Form-Key`:** при создании формы в CRM можно включить режим «защищённый
приём» (server-to-server). В этом случае форма хранит `ingest_token`, и каждый
запрос обязан содержать заголовок `X-Form-Key: <токен>`. Без него CRM вернёт `401`.

Для открытых форм (режим S2S выключен) ключ не нужен — запрос принимается от любого
источника, включая браузер.

> **Важно:** `X-Form-Key` — секрет. Никогда не включайте его в клиентский JavaScript
> на странице. Передавайте только с сервера сайта.

---

## 4. Тело запроса (JSON)

Тело — произвольный JSON-объект. CRM распознаёт следующие **канонические поля** и
их синонимы (регистр и разделители не важны):

| Канонический ключ | Принимаемые синонимы | Куда попадает |
|---|---|---|
| `company_name` | `company`, `name`, `название`, `название компании`, `наименование` | Название компании в карточке лида |
| `email` | `e-mail`, `почта` | Email лида |
| `phone` | `телефон`, `тел`, `mobile` | Телефон лида |
| `website` | `сайт`, `url` | Сайт компании |
| `city` | `город` | Город лида |
| `inn` | `инн` | ИНН компании |
| `comment` | `comments`, `message`, `комментарий`, `сообщение`, `вопрос` | Комментарий (Activity в карточке лида) |

**Нормализация ключей:** CRM переводит ключ в нижний регистр и заменяет `_` и `-`
на пробелы перед поиском в таблице синонимов. То есть `Company_Name`,
`company-name` и `company name` — одно и то же.

**Неизвестные ключи не теряются.** Если в теле есть поля, не вошедшие в список выше,
они сохраняются в `raw_payload` записи FormSubmission — менеджер может их просмотреть
в CRM. Это удобно для передачи кастомных атрибутов лендинга.

### UTM-параметры

Помимо полей формы, можно передать UTM прямо в теле запроса:

```
utm_source, utm_medium, utm_campaign, utm_content, utm_term
```

Embed-скрипт собирает их из `window.location.search` автоматически. При ручной
интеграции добавьте их сами из URL страницы перед отправкой.

### Пример тела

```json
{
  "company": "ООО Ромашка",
  "phone": "+7 900 000-00-00",
  "email": "info@romashka.ru",
  "comment": "Интересует кофейная станция на 3 точки",
  "utm_source": "yandex",
  "utm_medium": "cpc",
  "utm_campaign": "b2b-horeca-msk"
}
```

## 5. Ответы

| Код | Тело | Что означает |
|---|---|---|
| `200` | `{"ok": true, "redirect": "<url или null>"}` | Заявка принята, лид создан. Если `redirect` не `null` — перенаправьте пользователя на указанный URL (страница «спасибо»). |
| `401` | `{"detail": "Неверный или отсутствующий ключ формы"}` | Форма защищена, а `X-Form-Key` отсутствует или неверен. |
| `404` | `{"detail": "Форма не найдена"}` | Slug не существует. Проверьте настройки. |
| `410` | `{"detail": "Форма больше не принимает заявки"}` | Форма деактивирована в CRM. |
| `429` | `{"detail": "Слишком много заявок с этого IP. Попробуйте через минуту."}` | Превышен лимит запросов с одного IP. Повторите через ~60 секунд. |
| `500` | `{"detail": "Не удалось создать лид. Попробуйте позже."}` | Внутренняя ошибка CRM. Логируйте и уведомите команду DrinkX. |

## 6. Пример curl (S2S, с ключом)

```bash
curl -X POST https://crm.drinkx.tech/api/public/forms/<slug>/submit \
  -H "Content-Type: application/json" \
  -H "X-Form-Key: <ваш-ключ>" \
  -d '{
    "company": "ООО Ромашка",
    "phone": "+7 900 000-00-00",
    "email": "info@romashka.ru",
    "comment": "Интересует кофейная станция",
    "utm_source": "yandex"
  }'
```

Успешный ответ:

```json
{"ok": true, "redirect": null}
```

## 7. Пример Node.js (бэкенд сайта)

Паттерн: сначала отправляем письмо себе (как раньше), затем отправляем заявку в CRM.
Вызов CRM обёрнут в `try/catch` — недоступность CRM **не ломает** обработку формы
на сайте.

```js
// Пример для Express / Next.js API route (Node.js 18+)
// Предполагаем, что тело запроса уже распарсено в formData.

const CRM_SLUG = process.env.CRM_FORM_SLUG;          // например "main-landing"
const CRM_KEY  = process.env.CRM_FORM_KEY;            // секрет, только для S2S-форм
const CRM_URL  = `https://crm.drinkx.tech/api/public/forms/${CRM_SLUG}/submit`;

async function handleFormSubmit(formData) {
  // 1. Отправляем письмо на почту сайта — как и раньше
  await sendInternalEmail({
    to: "info@drinkx.ru",
    subject: `Заявка: ${formData.company || formData.name || "—"}`,
    body: JSON.stringify(formData, null, 2),
  });

  // 2. Передаём заявку в CRM — best-effort, не блокируем ответ пользователю
  try {
    const headers = { "Content-Type": "application/json" };
    if (CRM_KEY) headers["X-Form-Key"] = CRM_KEY;

    const res = await fetch(CRM_URL, {
      method: "POST",
      headers,
      body: JSON.stringify(formData),
      // Не ждём дольше 5 секунд
      signal: AbortSignal.timeout(5000),
    });

    if (!res.ok) {
      const body = await res.json().catch(() => ({}));
      console.warn("[CRM] Ошибка при передаче заявки", res.status, body.detail);
    }
  } catch (err) {
    // CRM недоступна — логируем, но не бросаем ошибку пользователю
    console.error("[CRM] Не удалось передать заявку:", err.message);
  }

  // 3. Возвращаем успех на фронтенд
  return { ok: true };
}
```

> Не добавляйте автоматические повторные попытки при `200` — CRM не гарантирует
> идемпотентность: повторная отправка одной заявки создаст второй лид.

## 8. Рекомендации

**Для форм с `X-Form-Key` (S2S):**
- Всегда отправляйте с бэкенда сайта. Ключ не должен появляться в HTML или клиентском JS.
- Храните ключ в переменных окружения сервера (`CRM_FORM_KEY`), не в репозитории.

**Для открытых форм (без ключа):**
- Можно отправлять напрямую из браузера через `fetch` — CORS разрешён для
  `/api/public/*`.
- Можно подключить готовый embed-скрипт — он сам отрисует форму, соберёт UTM
  из URL страницы и отправит данные:

  ```html
  <!-- Вставьте в то место страницы, где должна появиться форма -->
  <div id="drinkx-form-<slug>"></div>
  <script src="https://crm.drinkx.tech/api/public/forms/<slug>/embed.js" async></script>
  ```

  Скрипт монтируется в `#drinkx-form-<slug>` (или рядом с тегом `<script>`, если
  контейнер не найден). Кеш скрипта — 5 минут.

**Ретраи:**
- При `429` — подождите ~60 секунд, затем можно повторить.
- При `5xx` — логируйте и уведомляйте команду DrinkX; не ретраите в петле.
- При `200` — **не ретраить**: повторная отправка создаст дублирующий лид.

**Проверка подключения:**
- Тестируйте с реальным slug и данными — убедитесь, что лид появляется в CRM
  (раздел **Входящие → Пул**).
- Если лид не создался при статусе `200`, проверьте `raw_payload` записи в CRM —
  возможно, ключи полей не совпали ни с одним синонимом (имя компании тогда
  подставляется из домена Referer или имени формы).
