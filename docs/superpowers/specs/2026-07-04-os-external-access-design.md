# OS DrinkX → CRM: внешний read-only доступ (REST + MCP) — дизайн

Дата: 2026-07-04. Статус: спека утверждена на уровне архитектуры, реализация — отдельным этапом.

## 1. Цель

Дать «OS DrinkX» (внешняя система: собственный код + LLM-агенты) возможность
смотреть/читать/понимать данные CRM. Только чтение, только pull.

## 2. Зафиксированные решения

| Вопрос | Решение |
|---|---|
| Кто потребитель | Гибрид: программный код OS + LLM-агенты |
| Транспорт | REST-слой с машинным ключом + remote MCP поверх него |
| Уровень доступа | Только чтение (write — vNext, отдельный дизайн) |
| Идентичность | Один сервис-аккаунт (машинный ключ), без per-user прав |
| Данные | Ядро: лиды, компании, контакты, воронки/стадии, суммы, ответственные |
| Режим | Только pull (webhooks/события — vNext) |
| Формат | Сырые сущности + готовые LLM-сводки |
| MCP-хостинг | Remote MCP (streamable HTTP) на стороне CRM, crm.drinkx.tech |

## 3. Архитектура (выбран вариант A)

Новый доменный пакет `apps/api/app/external/` по стандартной структуре
package-per-domain (`models.py`, `schemas.py`, `repositories.py`, `services.py`,
`routers.py`). Роутеры монтируются под префиксом `/external/v1`. Пакет:

- имеет **собственную auth-зависимость** (API-ключ, не Supabase JWT);
- содержит **только GET-эндпоинты** — write-операций в пакете физически нет;
- **переиспользует** существующие репозитории (`leads`, `companies`, `contacts`,
  `pipelines`) для чтения, но отдаёт данные через **собственные Pydantic-схемы
  с явным whitelist полей** — никаких дампов внутренних моделей;
- MCP-эндпоинт монтируется на том же FastAPI-приложении и вызывает те же
  сервис-функции `external/services.py`.

Отвергнутые альтернативы:

- **B. Двойная аутентификация в существующих роутерах** — существующие роуты
  завязаны на пользователя и роль, высок риск случайно открыть лишнее, внешний
  контракт склеивается с внутренним.
- **C. Read-only Postgres-views через Supabase** — нет сводок, нет MCP на
  стороне CRM, контракт на уровне SQL хрупкий.

## 4. Аутентификация и ключи

- Заголовок: `Authorization: Bearer drinkx_os_<32+ случайных символа>`.
- Хранение: таблица `service_api_keys` (новая Alembic-миграция):
  `id`, `workspace_id` (FK — ключ привязан к workspace), `name`,
  `key_hash` (sha256 от полного токена), `scopes` (JSON, для OS: `["read:core"]`),
  `created_at`, `last_used_at`, `revoked_at`.
- Сам ключ показывается один раз при создании; в БД только хэш.
  Сравнение — constant-time (`hmac.compare_digest`).
- Выдача/отзыв v1 — CLI-скрипт или ручной INSERT (админ-UI не входит в scope).
- Ротация: создаём второй ключ, переключаем OS, отзываем старый
  (`revoked_at`) — без деплоя.
- Auth-зависимость `require_service_key(scope)` резолвит ключ → workspace_id,
  обновляет `last_used_at`, отклоняет отозванные и ключи без нужного scope.

## 5. REST-контракт `/external/v1`

Все эндпоинты — GET. Данные ограничены workspace ключа. Пагинация — курсорная
(`?cursor=...&limit=`, максимум 100), сортировка по `updated_at desc`.
Все даты в контракте — ISO 8601 UTC.

| Эндпоинт | Назначение | Параметры |
|---|---|---|
| `/leads` | список лидов | `pipeline_id`, `stage_id`, `assigned_to`, `updated_since` (ISO), `q` (поиск по названию компании), `cursor`, `limit` |
| `/leads/{id}` | карточка лида | — |
| `/leads/{id}/summary` | LLM-сводка одним вызовом | — |
| `/companies` | список компаний | `q`, `updated_since`, `cursor`, `limit` |
| `/companies/{id}` | карточка компании | — |
| `/contacts` | контакты | `lead_id` или `company_id` (один обязателен) |
| `/pipelines` | воронки со стадиями | — |
| `/pipelines/{id}/summary` | состояние воронки | — |
| `/meta` | версия контракта, справочники (стадии, источники, менеджеры-имена) | — |

Whitelist полей (первая версия):

- **Lead**: `id`, `company_name`, `segment`, `city`, `source`, `pipeline_id`,
  `stage_id`, `stage_entered_at`, `deal_amount`, `deal_quantity`,
  `deal_equipment`, `deal_type`, `priority`, `score`, `assignment_status`,
  `assigned_to` (+ имя менеджера), `next_action_at`, `last_activity_at`,
  `won_at`, `lost_at`, `tags`, `created_at`, `updated_at`.
  НЕ отдаём: `score_details_json`, `pilot_contract_json`, нормализованные
  служебные поля (`email_normalized`, `phone_e164`, `email_domain_criterion`).
  - `stage_entered_at` берётся из открытой строки `lead_stage_history`
    (`exited_at IS NULL`, LEFT JOIN — без N+1). Может быть `null`: история
    ведётся с миграции 0029 (2026-05-16), лид без смен стадии с тех пор
    строки не имеет. Семантика `null` = «неизвестно», OS не считает это датой.
  - `won_at`/`lost_at` — сырые поля модели; `closed_at` на стороне OS
    выводится как `coalesce(won_at, lost_at)`.
- **Company**: `id`, `name`, `inn`, `website`, `city`, `segment`, timestamps.
- **Contact**: `id`, `name`, `position`, `email`, `phone`, `lead_id`,
  `company_id`. (Контакты клиентов входят в «ядро» по решению пользователя.)
- **Pipeline/Stage**: `id`, `name`, `position`, `probability`, `is_won`,
  `is_lost`, `rot_days`.

Сводки:

- `/leads/{id}/summary` — компактный JSON: карточка лида (whitelist выше) +
  компания + контакты + текущая стадия (имя, probability) + возраст в стадии +
  флаги протухания (`is_rotting_*`). Одна цель: LLM-агент получает полную
  картину лида за один вызов.
- `/pipelines/{id}/summary` — по каждой стадии: количество лидов, сумма
  `deal_amount`, число протухших; итого по воронке.

Вне контракта (не отдаём в v1): переписка inbox/email, заметки, задачи,
таймлайн активности, КП, знания, данные пользователей CRM кроме имён
ответственных.

## 6. MCP-сервер

- Транспорт: streamable HTTP, путь `/mcp` на том же приложении
  (официальный python SDK `mcp`, mount как ASGI-приложение).
- Аутентификация: тот же Bearer-ключ, та же зависимость.
- Инструменты (все `readOnlyHint: true`, тонкие обёртки над
  `external/services.py`):
  - `search_leads(q, pipeline_id?, stage_id?, limit)` → компактный список;
  - `get_lead_summary(lead_id)` → как `/leads/{id}/summary`;
  - `pipeline_overview(pipeline_id?)` → как `/pipelines/{id}/summary`;
  - `list_pipelines()` → воронки со стадиями.
- Ответы компактные (без null-полей), под токен-бюджет агентов.

## 7. Ограничения, ошибки, наблюдаемость

- Ошибки: `401` — нет/битый ключ; `403` — отозван или нет scope;
  `404` — сущность не найдена в workspace ключа; `429` — rate limit.
  Формат: `{"detail": "..."}` (как в остальном API).
- Rate limit v1: 10 rps на ключ (in-memory token bucket в auth-зависимости;
  Redis-вариант — при появлении второй реплики API).
- Аудит: структурный лог каждого запроса (имя ключа, эндпоинт/инструмент,
  latency, статус) через существующий логгер; Sentry ловит ошибки как обычно.
- Версионирование: префикс `/v1`, изменения только аддитивные; слом контракта —
  только через `/v2`.

## 8. Тестирование (критерии готовности реализации)

1. Auth: валидный ключ → 200; битый/отсутствующий → 401; отозванный → 403;
   ключ чужого workspace не видит данные (404 на чужой lead_id).
2. Снапшот-тест: под `/external` не зарегистрировано ни одного не-GET роута.
3. Схемы: в ответах нет полей вне whitelist (тест сериализации).
4. Сводка лида собирается за ≤ 3 SQL-запроса и укладывается в схему.
5. MCP: `tools/list` отдаёт 4 инструмента, `get_lead_summary` совпадает по
   содержимому с REST-сводкой.
6. `/leads` отдаёт `stage_entered_at` из открытой строки `lead_stage_history`
   одним запросом (без N+1); лид без истории → `null`.

## 9. Вне скоупа (vNext)

- Запись (создание заметок/задач из OS) — отдельный дизайн с идемпотентностью
  и human-in-the-loop.
- История переходов стадий наружу (`/leads/{id}/stage-history`) — данные уже
  есть (append-only `lead_stage_history`, миграция 0029), эндпоинт тривиален;
  первый кандидат на v1.1 по запросу OS.
- Webhooks / события в OS; `updated_since` уже покрывает дешёвый поллинг.
- Per-user identity и соблюдение ролей head/manager для OS.
- Переписка, активность/таймлайн, КП, аналитика менеджеров.
- Админ-UI управления ключами.

## 10. План реализации (эскиз, детальный план — отдельно)

1. Миграция `service_api_keys` + auth-зависимость + CLI выдачи ключа.
2. Пакет `external`: схемы, сервисы, роутеры `/external/v1/*`, тесты.
3. MCP-mount `/mcp` с 4 инструментами, тесты.
4. Прогон против прода-стейджа, выдача первого ключа OS, короткий README
   для стороны OS (базовый URL, заголовок, примеры вызовов).

## 11. Профиль потребления стороны OS (принято к сведению, 2026-07-04)

OS-сторона (Research Brain) приняла контракт v1; её три аддитивных запроса
(`stage_entered_at` в списке лидов, `won_at`/`lost_at`, фиксация ISO 8601 UTC)
включены в §5 выше. Как OS будет потреблять контракт:

- Ночной/ежечасный pull по `updated_since` с вотермаркой, идемпотентный ингест
  снапшотами → лимит 10 rps из §7 заведомо достаточен.
- Сырые поля сущностей OS использует как цитируемые факты (evidence);
  LLM-сводки (`/leads/{id}/summary`) — только как контекст. Значит контракт
  сырых полей — самая стабильная часть API: менять только аддитивно.
- Имена ответственных из `/meta` мапятся на реестр сотрудников OS —
  формат «имя менеджера» в `/meta` должен быть стабильным.
- Ключ OS хранит в gitignored `.env`, в логи не пишет.
