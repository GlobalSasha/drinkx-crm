# Дизайн: «Обновление базы лидов» (`base_update`)

**Дата:** 2026-05-21
**Статус:** утверждён к реализации
**Домен:** новый `apps/api/app/base_update/` + секция в `apps/web` (Настройки)

## Проблема

Менеджеры/ассистент готовят насыщенные `.md` карточки ЛПР (компания + несколько
контактов с ролями + аналитический бриф). Нужен повторяемый инструмент: загрузить
пачку `.md` → CRM сама их анализирует и корректно вносит в базу, не плодя дубли и не
затирая ручные правки менеджеров.

Существующий домен `import_export` (CSV/Excel/JSON/Bitrix24/AmoCRM, preview→apply) для
этого не подходит: он только **создаёт лиды по строкам**, без контактов, AI-брифа,
дедупликации и обновления существующих. Поэтому — отдельный домен.

## Требования (зафиксированы при брейншторме)

- **Вход:** `.md` файлы (формат «плавает»), CRM анализирует их сама.
- **Извлечение:** AI (не жёсткий парсер) — формат карточек нестабилен.
- **Запись:** автоматическая для безопасного; спорное удерживается как конфликт на ревью.
- **Дедуп / «добавлять только недостающее»:** не перезаписываем заполненные поля, не
  удаляем контакты — только дополняем. Расхождения → конфликты.
- **Новые лиды:** в пул, с флагом `needs_review` (менеджер потом дочищает в карточке).
- **Доступ:** admin/head, секция в «Настройках» рядом с обычным импортом.
- **Перенос из карточки:** компания + все ЛПР как контакты с ролями + AI-бриф.

## Архитектура и поток данных

```
1. Загрузка     админ кидает N .md → POST → IngestJob (202), файлы во временном хранилище
2. Извлечение   Celery: по файлу LLM-экстрактор → ExtractedCard {company, contacts[], ai_brief}
3. Дедуп пачки  слить карточки об одной компании; расхождения/похожие имена → конфликт #6
4. Матчинг      нормализация имени → 0 совпадений: «создать»; 1: «дополнить»; >1: конфликт #1
5. Дифф         поля/контакты/бриф → «авто-применить» vs «конфликты #2–#5»
6. Авто-запись  безопасное пишется сразу (сервисы companies/leads/contacts); новые лиды → пул + needs_review
7. Конфликты    остаются в job как IngestConflict → админ разрешает по одному
8. Применение   решения админа применяются (keep/overwrite/manual/add/pick/skip)
```

**Переиспользование (без дублирования логики):**
- извлечение → `enrichment.providers.factory.complete_with_fallback` (MiMo Flash → Anthropic → Gemini → DeepSeek)
- нормализация имени компании → `companies.utils.normalize_company_name`
- запись → доменные сервисы `companies` / `leads` / `contacts`
- бюджет/стоимость LLM → `enrichment.budget` + `llm_usage`

## Модель данных

`IngestJob 1—N IngestRecord 1—N IngestConflict`. Всё workspace-scoped, FK cascade.

### IngestJob — одна загрузка-пачка
- `id`, `workspace_id`, `user_id`
- `status`: `pending → extracting → matching → ready` (авто-применено, конфликты ждут) `→ resolving → done / failed`
- `source_filenames` (JSON), `file_count`
- `stats_json`: компаний создано / лидов создано / контактов добавлено / полей заполнено / конфликтов всего / открыто
- `error`, `created_at`, `finished_at`

### IngestRecord — одна компания из пачки (после дедупа) + план
- `id`, `ingest_job_id` (FK cascade)
- `company_name`, `normalized_name`
- `extracted_json` — полный AI-результат `{company, contacts[], ai_brief}`
- `match_company_id` / `match_lead_id` (nullable)
- `action`: `created | updated | conflict | skipped`
- `source_files` (JSON), `confidence`, `error` (nullable)

### IngestConflict — один конфликт под решение
- `id`, `ingest_job_id`, `ingest_record_id` (FK)
- `type`: `company_ambiguous | field_mismatch | contact_mismatch | lead_target | low_confidence | batch_duplicate` (1–6)
- `target_kind`: `company | lead | contact | brief`, `field_name` (nullable)
- `base_value` / `incoming_value` / `candidates_json` (для #1, #4)
- `status`: `open | resolved | skipped`
- `resolution`: `keep | overwrite | manual | add_separate | pick | skip` + `resolved_value` (для manual/pick)
- `resolved_by`, `resolved_at`

## AI-извлечение

Один LLM-вызов на файл через `complete_with_fallback`, новый `TaskType.lpr_extraction`.
Стоимость → `llm_usage`; перед вызовами — проверка дневного бюджета.

Схема вывода (Pydantic, всё `Optional` + дефолты, **никогда не бросает**):
```
ExtractedCard:
  company:  name (обяз.), segment, priority(A/B/C/D), website, inn, city, phone, email
  contacts: [ {name, title, role_type, email, phone, telegram, linkedin, source, confidence} ]
  ai_brief: str            # описание + масштаб + кофейный сервис + триггеры + маршрут
  extraction_confidence: float (0–1)
```

Промпт: извлечь структуру из русской markdown-карточки; **роль ЛПР маппит LLM** в один из
4 канонических типов (`economic_buyer`/`champion`/`technical_buyer`/`operational_buyer`) с
определениями ролей в промпте; чего нет — `null`, не выдумывать.

Триггеры конфликта #5: `extraction_confidence` ниже порога, пустое `company.name`, либо
контакт без имени / с битым email.

## Матчинг, дедуп и правила авто-применения

**Дедуп пачки (#6):** группировка по `normalize_company_name`. Точное + согласовано →
слить молча. Триграммно похожи (имя разное) или поля расходятся → конфликт #6.

**Матчинг с базой:** 0 совпадений → «создать» (компания + лид в пул + `needs_review`);
1 → «дополнить»; >1 (точное + триграммные) → конфликт #1 со списком кандидатов.

**Куда привязать (#4):** у компании 0 лидов → создать лид; 1 → дополнить; >1 → конфликт #4.

**Поля (#2)** (`segment, priority, website, inn, city, email, phone`): пусто в базе →
авто-заполнить; одинаково → ничего; заполнено и отличается → конфликт #2.

**Контакты:** матч по нормализованному имени в рамках компании/лида. Нет совпадения →
авто-добавить (`verified_status=to_verify`); имя совпало, детали отличаются → конфликт #3;
полное совпадение → ничего.

**AI-бриф:** нет брифа у лида → авто-записать; есть → не трогать, отдать конфликт #2 на бриф.

**Незыблемо:** заполненные поля не перезаписываем, контакты не удаляем — только дополняем.

## Экран (Настройки, admin/head)

1. **Загрузка:** drag-and-drop `.md` → «Загрузить и разобрать» → job, прогресс
   (`извлечение → матчинг`), фронт поллит статус (паттерн обогащения).
2. **Сводка:** счётчики created/updated/contacts/fields + число конфликтов.
3. **Конфликты по одному.** Карточка: «В базе … / Из карточки …» + кнопки под тип:
   - #2/#6 поле: Оставить базу · Взять из карточки · Ввести вручную
   - #1 компания: выбрать кандидата · Создать новую · Пропустить
   - #4 лид: выбрать лид · Создать новый лид
   - #3 контакт: Обновить · Добавить отдельно · Оставить базу · Пропустить
   - #5 низкая уверенность: Исправить вручную · Пропустить
   - #6 дубль пачки: Это одна компания (слить) · Разные (создать обе)
4. **«Применить решения»** → запись через доменные сервисы. Открытых конфликтов 0 → `done`.

Вёрстка — в фирменном стиле приложения (brand-токены, светлая тема) на этапе реализации.

## API (`/api/base-update/*`, admin/head)

- `POST /jobs` — multipart `.md` → IngestJob + Celery → 202 `{job_id}`
- `GET /jobs/{id}` — статус + stats
- `GET /jobs/{id}/conflicts` — список (open/all)
- `PATCH /conflicts/{id}` — `{resolution, resolved_value}`
- `POST /jobs/{id}/apply` — применить решённые (Celery) → 202
- `GET /jobs` — история (пагинация)

## Асинхронность

Как у обогащения/импорта: Celery-задачи с `NullPool`-движком на вызов
(`_build_task_engine_and_factory`), регистрация в `celery_app`.
- `extract_and_match(job_id)` — извлечение по файлам, дедуп, матчинг, **авто-запись
  безопасного**, статус `ready`.
- `apply_resolutions(job_id)` — применить решённые конфликты, статус `done`.

## Обработка ошибок (best-effort, не роняем всю пачку)

- Сбой извлечения файла → запись с конфликтом #5, остальные файлы продолжают.
- Бюджет LLM исчерпан → стоп извлечения, job частичный с причиной в `error`.
- Сбой записи по компании → в `IngestRecord.error`, job идёт дальше.
- `audit.log` на created/updated; workspace-изоляция в каждом запросе.

## Тестирование

- **Юнит:** разбор схемы (дефолты на кривом вводе), матчер (0/1/>1 компания, выбор лида),
  дедуп пачки (#6), классификация поле→авто/конфликт, маппинг ролей. Фикстуры — реальные
  `.md` карточки из `~/Desktop/DrinkX_Retail_LPR` и `DrinkX_AZS_LPR`.
- **Интеграция:** end-to-end на тестовой БД с замоканным LLM (upload → extract →
  records/conflicts → resolve → apply).
- **Фронт:** `pnpm build` + `lint` зелёные; e2e не требуется на первом этапе.

## Ограничения / вне скоупа

- Принимаем только `.md`; потолок на число и размер файлов (конфиг).
- Не синхронизация (одноразовые загрузки), не источники-коннекторы — это отдельная фича.
- Разовый скрипт `scripts/lpr_import/` остаётся как черновой инструмент, но не часть продукта.

## Заметки по корректности данных (из брейншторма)

- Дикси — самостоятельный лид, **не часть X5**; ошибочная привязка к `purchase@x5.ru` из
  исходных карточек не переносится (см. память `lead-data-diksi-x5`).
- Газпромнефть/Лукойл/Роснефть встречаются в обеих исходных папках под разными именами —
  дедуп пачки (#6) обязан их ловить.
