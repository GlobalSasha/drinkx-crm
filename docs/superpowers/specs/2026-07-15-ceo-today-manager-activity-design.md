# Спека: `/today` руководителя — панель работы менеджеров

**Дата:** 2026-07-15
**Статус:** утверждён макет, ждёт ревью спеки
**Автор:** сессия Claude (бренч `polish/motion` → отдельная ветка на реализацию)

## 1. Зачем

Руководитель (`role` = `head`/`admin`) не ведёт свою воронку — он смотрит на
**результаты менеджеров по продажам**. Сейчас его `/today` показывает
`CeoOverview` = обзор **потока заявок** (сколько лидов, откуда, что застряло).
Это не отвечает на вопрос «кто из менеджеров что делает или не делает».

Экран нужно развернуть с «потока лидов» на «работу людей»: **сколько менеджер
реально работал в CRM и что в итоге с его лидами** — в разрезе день / неделя / месяц.

Сейчас в команде один менеджер, скоро будет больше — экран обязан хорошо
выглядеть и для одного (крупная карточка), и для многих (таблица-строки).

## 2. Что показываем (макет утверждён)

Экран `/today` для `head`/`admin`:

```
Заголовок «Команда»            [ Сегодня · Неделя · Месяц ]  ← один переключатель
⚠️ Полоса тревог (только если есть повод)
── если 1 менеджер → большая карточка «ТРУД ↔ РЕЗУЛЬТАТ»
── если много → та же метрика в таблице-строках, сортировка по столбцу
```

**Блок ТРУД (труд менеджера за период):**
- Активное время в CRM — крупная цифра + столбики по дням (14 дней) *(новый замер)*
- Новых лидов добавил
- Действий всего (касания/обновления данных)
- КП отправлено

**Блок РЕЗУЛЬТАТ (что с его лидами):**
- В работе сейчас
- Продвинул по воронке (смен этапа за период) — это и есть показатель движения
- Застряло 7+ дней (+ красный флаг)
- Задач закрыто / просрочено
- Был активен (когда последний раз)

Утверждённый макет-прототип: `scratchpad/ceo-today-mockup.html` (интерактивный,
на фирменных токенах). Скриншоты обоих режимов согласованы в чате.

## 3. Как считаем (определения метрик — залочены)

Все агрегаты **workspace-scoped**, read-only. Период → диапазон `[from_, now]`:
- `day` = сегодня с 00:00 UTC
- `week` = 7 дней (сегодня−6 дней, с 00:00)
- `month` = 30 дней (сегодня−29 дней, с 00:00)

(Согласовано с существующим `company/services._period_start`; добавляем `day`.)

| Метрика | Как считаем | Источник |
|---|---|---|
| **Активное время** | Кол-во уникальных «минут активности» в диапазоне × 1 мин. Для спарклайна — группировка по дню. | новая таблица `presence_minutes` |
| **Новых лидов добавил** | `leads`: `assigned_to = u AND created_at ∈ период AND archived_at IS NULL`. *(У лида нет `created_by` — это лучший прокси: «новые лиды на менеджере».)* | `leads` |
| **Действий всего** | `activities`: `user_id = u AND created_at ∈ период AND type IN (comment, task, reminder, file, email, tg, phone, form_submission)`. Системные/AI типы (`system, ai_suggestion, lead_assigned, enrichment_done, score_update, stage_change`) не считаем — это «касания человека». | `activities` |
| **КП отправлено** | `activities`: `user_id = u AND type='file' AND file_kind='kp' AND created_at ∈ период`. *(как в `team/repositories.kp_sent_per_user`)* | `activities` |
| **Продвинул по воронке** | `audit_log`: `user_id = u AND action='lead.move_stage' AND created_at ∈ период`. *(как в `team.leads_moved_per_user`)* | `audit_log` |
| **Задач закрыто** | `activities`: `user_id = u AND type='task' AND task_done=true AND task_completed_at ∈ период`. *(как в `team.tasks_completed_per_user`)* | `activities` |
| **Задач просрочено** | `activities`: `type='task' AND task_done=false AND task_due_at < now`, где лид (`activities.lead_id`) назначен на `u`. Точечная величина (не за период). | `activities`+`leads` |
| **В работе сейчас** | активные назначенные лиды на нестерминальной стадии: `assigned_to=u, assignment_status='assigned', archived_at IS NULL, stage.is_won=false, is_lost=false`. *(как `team.portfolio_kpi.active_count`)* | `leads`+`stages` |
| **Застряло 7+ дней** | из «в работе» те, у кого `is_rotting_stage OR is_rotting_next_step`. `oldest_stuck_days` = макс. дней с `last_activity_at`. *(канонические флаги «устаревания», как виджет «Устаревает» на `/today`)* | `leads` |
| **Был активен** | `max(activities.created_at)` по `u`, фоллбэк на `users.last_login_at`. *(как `team.last_active_per_user` + фоллбэк в сервисе)* | `activities`/`users` |

> **Убрано (решение 2026-07-15):** «Конверсия его лидов» (% дошедших до квалификации).
> При малом числе лидов процент волатилен, а «Продвинул по воронке» + «Застряло» уже
> честно показывают движение и застой. Показатель движения = смены этапа, отдельного
> процента не вводим.

**Тревоги (считаются на сервере, показываем только непустые):**
- `silent` — `был активен` старше 48 ч, при этом в workspace есть лиды → «{Имя} не заходил N дней».
- `stuck` — у менеджера `застряло > 0` → «N лидов застряли 7+ дней у {Имя}».

## 4. Как считаем «активное время» (новый замер)

**Определение:** активная минута = минута, в которой у менеджера была **живая
работа** (мышь/клавиши/скролл/переходы) и вкладка была видима. Не «вкладка открыта».

**Фронт — хук `use-presence-heartbeat.ts`** (монтируется в авторизованном шелле
`app/(app)/layout.tsx`):
- Слушает `pointerdown, keydown, scroll, touchstart, mousemove(throttle ~5с), visibilitychange`; держит `lastInteractionAt`.
- Раз в 60 с: если `document.visibilityState === 'visible'` И `now - lastInteractionAt ≤ 60_000` → `POST /api/presence/ping` (без тела).
- Один пинг на мгновение монтирования (если видима), дальше не чаще 1/мин.
- Простаивающая вкладка (нет активности > 60с) пинг не шлёт → минута не идёт в зачёт.

**Бэк — новый домен `app/presence/`:**
- `POST /api/presence/ping` (auth). Сервер: `minute = date_trunc('minute', now())`,
  `INSERT (user_id, workspace_id, minute) ON CONFLICT DO NOTHING`.
  Дедуп по нескольким вкладкам и повторным пингам — за счёт PK.
- Таблица `presence_minutes`:
  - `user_id UUID FK users(id) ON DELETE CASCADE`
  - `workspace_id UUID NOT NULL` (денормализовано для быстрых workspace-range запросов)
  - `minute TIMESTAMPTZ NOT NULL` (усечено до минуты, UTC)
  - `PRIMARY KEY (user_id, minute)`
  - `INDEX (workspace_id, minute)`
- Агрегаты:
  - `active_minutes_range(workspace_id, user_ids, from_, to)` → `{user_id: count}`
  - `active_daily(workspace_id, user_id, from_, to)` → `[{date, minutes}]` (спарклайн, 14 дней)

**Объём данных:** ~1 строка на активную минуту на пользователя (≈480/день при 8ч).
Для команды в единицы человек — незначительно. Ночной роллап — на будущее (YAGNI сейчас).

**Честная оговорка (в UI подпись):** замер копит данные **только с момента выката**.
В первый день «активное время» ≈ 0, «за неделю» наберётся за неделю, «за месяц» — за
месяц. Задним числом не восстановить. Все остальные метрики показываются сразу (данные уже есть).

## 5. API

**`GET /api/company/managers?period=day|week|month`** (расширяем домен `app/company/`).

Ответ:
```jsonc
{
  "period": "week",
  "totals": { "active_minutes": 0, "new_leads": 0, "actions": 0, "kp_sent": 0, "stuck": 0 },
  "alerts": [ { "type": "stuck", "user_id": "…", "name": "Иван", "count": 3 } ],
  "managers": [
    {
      "user_id": "…", "name": "Иван Петров", "role": "manager",
      "active_minutes": 760, "active_daily": [ { "date": "2026-07-09", "minutes": 45 }, … ],
      "new_leads": 9, "actions": 57, "kp_sent": 4,
      "stage_moves": 6, "tasks_done": 11, "tasks_overdue": 2,
      "in_work": 18, "stuck": 3, "oldest_stuck_days": 12,
      "last_active_at": "2026-07-15T08:40:00Z"
    }
  ]
}
```

**Ростер менеджеров** = пользователи workspace с `role = 'manager'` (LEFT JOIN метрик,
чтобы менеджер с нулём активности всё равно показывался — это и есть «кто не работает»).
Сортировка по умолчанию — по `active_minutes` desc (клиент может пересортировать).

**Переиспользуем** существующие агрегаты из `app/team/repositories.py` (kp/moved/tasks/
last_active/portfolio) — вызываем их из company-сервиса, не дублируем SQL. Новое: presence-
агрегаты, `new_leads` (created_at-based), `actions` (список типов), `tasks_overdue`,
сборка тревог.

## 6. Фронт

- Новый компонент `components/today/TeamOverview.tsx`, рендерится `TodayRouter` для
  `head`/`admin` вместо нынешнего `CeoOverview`.
- Один `useState<'day'|'week'|'month'>('week')` → один запрос `useCompanyManagers(period)`
  (`lib/hooks/use-company.ts`). **Один источник правды** для периода — в прошлый раз
  тоггл рассинхронился и его сняли; здесь не повторяем.
- `managers.length === 1` → карточка-спотлайт (2 колонки труд/результат, спарклайн из
  `active_daily`). `> 1` → таблица-строки с сортировкой. `0` → пустое состояние.
- Только фирменные токены (`C`, `type-*`, `Card`, `Badge`, палитра графиков). Пилюльные
  контролы, крупные скругления, кремовый канвас — как в макете.
- Тревоги — компактная полоса на подложке `brand-soft`, рендерим только при непустом `alerts`.
- Клик по строке (много менеджеров): v1 — inline-разворот в тот же спотлайт. Навигация в
  глубокую аналитику `/team` — как переиспользование, опционально (P2).

## 7. Что со старым обзором потока заявок

Нынешний `CeoOverview` (источники, конверсия с рекламы, график по дням) полезен CEO, но
это уже не главный экран. **Решено (2026-07-15): вариант «б»** — оставляем свёрнутой
секцией «Поток заявок» под таблицей менеджеров. Существующий контент `CeoOverview`
(источники, конверсия с рекламы, график по дням) выносим в переиспользуемый компонент и
оборачиваем в аккордеон (свёрнут по умолчанию) под блоком менеджеров — компонент уже готов,
ценность сохраняется, работы минимум.

`/team` (глубокая аналитика по менеджеру) пока **остаётся как есть**; таблица `/today`
позже может линковаться в неё. Консолидация `/team` → отдельный будущий шаг (не в этом спринте).

## 9. Границы (YAGNI)

- Без ночного роллапа presence (добавим при росте объёма).
- Без AI-дайджеста, без прогресса к квоте, без выручки наверх (сделки редки/долги).
- «Новых лидов добавил» — по `assigned_to`+`created_at` (нет `created_by`); если позже
  появится авторство создания — уточним.
- Клик-разворот строки и консолидация `/team` — минимально/на будущее, не блокируют v1.
