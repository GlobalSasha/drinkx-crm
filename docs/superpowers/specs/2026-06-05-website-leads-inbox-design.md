# Раздел «Входящие заявки» (Website Leads Inbox)

**Дата:** 2026-06-05
**Статус:** утверждён (визуальный макет одобрен заказчиком)

## Проблема

Заявки с сайтов/форм сейчас видны только в общей «Базе лидов», вперемешку со
всеми лидами. Заказчику нужен **отдельный раздел меню** только для входящих
заявок с форм: со счётчиком новых (как непрочитанные письма), всплывающим
уведомлением и быстрым переходом в карточку лида.

## Что уже есть (переиспользуем)

- `FormSubmission` — каждая заявка сохраняется (lead_id, web_form_id, raw_payload,
  utm, source_domain, created_at).
- Уведомления «Новая заявка с сайта» уже шлются (`_notify_workspace_admins`).
- Навигация поддерживает бейдж-счётчик (`badge?: number`, как у «Уведомлений»).
- Опрос бейджа уведомлений (`useNotificationsBadge`) — образец для inbox-бейджа.

## Модель «новое / просмотрено» (v1)

Простейшая модель «как непрочитанная почта», по одному пользователю:
- Новая колонка `users.forms_inbox_seen_at` (timestamptz, nullable).
- **Бейдж** = число `form_submissions` воркспейса с `created_at > seen_at`
  (или все, если `seen_at` пуст).
- Элемент «новый», если `created_at > seen_at`.
- Открытие раздела → фронт вызывает «отметить просмотренными» (seen_at = now),
  бейдж обнуляется. Кнопка «Отметить все прочитанными» делает то же явно.

Пер-итемный read (открыл одну заявку → она перестала быть новой) сознательно
отложен: модель last_seen покрывает 90% ценности без join-таблицы. YAGNI.

## Backend

Все эндпоинты — в существующем `app/forms/routers.py` (authed), объявлены ДО
параметрических `/{form_id}` маршрутов (как `/analytics`), чтобы строковый
сегмент `inbox` не парсился как UUID.

- `GET /api/forms/inbox?form_id=&unseen_only=&page=&page_size=`
  → страница элементов inbox: по одному на `FormSubmission`, с присоединёнными
  данными лида (company_name, phone, email, assignment_status, assigned_to →
  имя менеджера, stage_id) и формы (name, source_label, slug). Плюс:
  - `snippet` — короткий текст обращения из `raw_payload` (ключи
    comment/message/сообщение), обрезанный.
  - `is_new` — `created_at > caller.seen_at`.
  - Сортировка: новые сверху (`created_at DESC`).
- `GET /api/forms/inbox/badge` → `{ "new": <int> }` (для опроса в меню).
- `POST /api/forms/inbox/seen` → `seen_at = now()`, возвращает `{ "new": 0 }`.

Repository: один запрос с JOIN `form_submissions ⋈ web_forms ⋈ leads`,
фильтр по `web_forms.workspace_id = caller.workspace_id`. Имя менеджера —
доп. join на `users` или резолв по `assigned_to`.

Миграция `0047_user_forms_inbox_seen`: `ALTER TABLE users ADD COLUMN
forms_inbox_seen_at timestamptz NULL`.

Pydantic-схемы: `InboxItemOut`, `InboxPageOut`, `InboxBadgeOut`. Все поля
Optional с дефолтами (конвенция: схемы не падают на пустых данных).

## Frontend

- Маршрут `app/(app)/inbox/page.tsx` — лента + фильтр-чипы по источнику +
  «Только новые» + превью справа (детали заявки) + переходы.
- Пункт меню «Входящие» в `SidebarNavContainer` между «Воронкой» и «Базой
  лидов», с бейджем из нового хука `useInboxBadge` (опрос как у уведомлений).
- Хуки в `lib/hooks/use-inbox.ts`: `useInbox` (список), `useInboxBadge`,
  `useMarkInboxSeen`.
- При открытии раздела (после загрузки списка) вызывается `useMarkInboxSeen`,
  чтобы обнулить бейдж.
- «Открыть карточку лида» → `/leads/[id]`. «Взять в работу» → существующий
  claim-флоу (если найдётся переиспользуемый хук; иначе кнопка ведёт в лида).
- Превью справа на узких экранах сворачивается (mobile-first): на мобиле клик
  по заявке открывает карточку лида напрямую.

Всплывающее уведомление при новой заявке — отдельная маленькая итерация: на v1
достаточно бейджа + существующего колокольчика «Новая заявка с сайта». Тост
можно добавить позже поверх существующего notifications-стрима.

## Anti-scope (НЕ делаем в v1)

- Пер-итемный «прочитано».
- Тост-уведомление реального времени (бейдж + колокольчик достаточно).
- Действия массового назначения/архивации из inbox.

## Проверка

- Backend: `py_compile` + новые юнит-тесты (snippet-экстрактор, badge-подсчёт,
  гейтинг seen). pytest — в CI.
- Frontend: `npm run typecheck` + `pnpm build` (обязательно — новый маршрут).
- Миграция применяется автоматически (`alembic upgrade head` в Dockerfile).
- Ручная проверка после деплоя: заявка с сайта → бейдж +1 → раздел «Входящие» →
  заявка в ленте → «Открыть карточку лида».
